1. Comment ça marche

L'application repose sur trois briques indépendantes, à relier entre elles :

  • Le document Grist — la base de données (étudiants, stages, plannings, cadres…) ;
  • Le Worker Cloudflare (dossier worker/) — un proxy qui interroge Grist à la place du site et garde la clé API secrète ;
  • Le site (dossier docs/, publié sur GitHub Pages) — les espaces étudiant et cadre, qui ne parlent qu'au Worker.

Déployer sur un autre établissement = copier le document Grist, déployer un Worker qui pointe dessus, et publier le site avec l'URL de ce Worker. Cette page génère pour vous les deux fichiers de liaison (worker/wrangler.toml et docs/config.js) à partir de vos liens.

Prérequis : un compte Grist (par ex. Grist DINUM), un compte Cloudflare (gratuit), un compte GitHub et Node.js sur votre poste. Le pas-à-pas complet est dans INSTALL.md à la racine du dépôt.

2. Préparer le document Grist

  1. Depuis le document d'origine : menu Fichier → Faire une copie, vers votre organisation/espace Grist. Toutes les tables, colonnes et formules sont copiées.
  2. Dans la copie, videz les données (LISTE_DES_ETUDIANTS, PERIODES_DE_STAGE, PLANNING_HEBDO, Sortie_de_stage, EVALUATION_STAGE_ETUDIANT) et adaptez les références (SERVICES, UTILISATEURS, CODES_HORAIRES, JOURS_FERIES, ETABLISSEMENT…).
  3. Récupérez une clé API Grist : avatar en haut à droite → Paramètres du profil → Clé API. Le compte doit avoir accès en écriture au document.
La clé API ne se saisit nulle part sur ce site. Elle ne doit jamais apparaître dans le code ni dans un fichier du dépôt : elle sera enregistrée comme secret Cloudflare à l'étape 6 (npx wrangler secret put GRIST_API_KEY).

5. Liens internes à Grist

        
Le document Grist contient lui aussi des liens vers le site. Plutôt que
          de les laisser codés en dur dans chaque formule, ils sont
          centralisés dans la table ETABLISSEMENT
          (celle qui porte déjà le nom et le logo affichés sur le site) : les
          formules lisent la table, et changer une valeur dans la table met à
          jour tout le document automatiquement.


        
a. Colonnes à ajouter à la table ETABLISSEMENT

        
Quatre colonnes de type Texte (données, pas formules),
          sur la première ligne de la table. Url_document_grist
          alimente le lien « Administration (Grist) » du pied de page du site,
          via l'endpoint /api/config du Worker :

        

          

            ETABLISSEMENT — valeurs à saisir
            reprises des champs de l'étape 3
            
          

          

        


        
b. Formules à remplacer — une seule fois

        
Ces formules ne contiennent plus aucune URL : elles
          sont identiques pour tous les établissements. Faites le remplacement
          une seule fois (dans le document d'origine ou dans la copie) ; ensuite,
          chaque nouveau déploiement n'a plus qu'à remplir la table
          ETABLISSEMENT. Pour chaque colonne : ouvrez la
          table, sélectionnez la colonne, panneau latéral « Colonne » → formule.

6. Commandes à exécuter

        
Dans un terminal, à la racine de votre copie du dépôt (une fois
          wrangler.toml mis à jour) :


        

          

            Terminal
            déploie le Worker (proxy)
            
          

          
cd worker
npm install
npx wrangler login                     # ouvre le navigateur pour autoriser
npx wrangler secret put GRIST_API_KEY  # collez ici la clé API Grist (étape 2)
npx wrangler deploy                    # affiche l'URL du Worker

        


        
npx wrangler deploy affiche l'URL du Worker :
          reportez-la dans le champ « URL du Worker » de l'étape 3, mettez à jour
          docs/config.js avec le contenu régénéré, puis
          poussez le dépôt sur GitHub et activez GitHub Pages
          (Settings → Pages → branche main, dossier
          /docs).

      


      

        
7. Tester la connexion

        
Vérifie que le Worker répond et qu'il accède bien à votre document
          Grist (endpoints publics uniquement, aucune donnée nominative).

        

        

        

          Le test échoue en « erreur réseau » ? Le Worker n'accepte
          que les appels venant de l'origine déclarée dans
          ALLOWED_ORIGIN. Si vous ouvrez cette page depuis
          une autre adresse (poste local, autre site), le navigateur bloque l'appel :
          refaites le test depuis cette même page une fois le site publié
          à l'adresse renseignée à l'étape 3, ou vérifiez l'URL du Worker.
